# Calibration Procedure

This procedure mines the user's own Claude Code conversation history for personal pptx correction patterns and adds them as custom checks in `~/.claude/skills/slide-inspector/user-overrides.yaml`.

Calibration is **opt-in**. Run it when the user invokes `calibrate slide-inspector` (or similar).

## When to Run

Recommend calibration only if:
- The user has at least ~10 sessions where they used the pptx skill or slide-inspector
- The user wants their personal style codified as custom checks
- The user wants to capture corrections that go beyond the defaults

If the user has very little pptx history, calibration won't find meaningful patterns. Tell them to come back after more sessions.

## Procedure

### Step 1: Pre-filter (bash)

Run the pre-filter helper to extract candidate correction messages from the user's CC history:

```bash
bash <this-skill-directory>/scripts/calibrate.sh > /tmp/slide-inspector-calibration-corpus.txt
```

This script:
- Iterates over `~/.claude/projects/*/` directories
- Greps each `.jsonl` file for pptx-related keywords (pptx, powerpoint, slide-inspector, pptxgenjs, edu-workshop-slides)
- For matching files, uses `jq` to extract user-role messages
- Filters those messages with a correction-language heuristic (English: change/fix/instead/wrong; Russian: переделай/не нравится/неправильно/почини)
- De-dupes near-identical messages (Bayram retries on network failures, would inflate counts)
- Outputs one candidate correction per line, prefixed with project + session ID

The output is a corpus of ~100-300 candidate correction messages.

### Step 2: Cluster patterns (Claude judgment)

Read the corpus file. Cluster messages by underlying intent. Look for:
- Layout fixes (move X, resize Y) — programmatic check candidates
- Style fixes (font, color, spacing) — possibly a custom regex
- Content fixes (this number is wrong, add citation) — content discipline
- Cross-slide fixes (make consistent across slides) — cross-slide check
- AI-output complaints (too verbose, too generic, AI-sounding) — text-content check
- Domain-specific (e.g., "the character on slide 5 is wrong") — only useful if there's a verifiable rule

For each cluster:
- Count rough frequency (number of messages in the cluster)
- Pick 1-2 verbatim examples
- Determine if it's already covered by an existing default check (read SKILL.md)

### Step 3: Triage with the user

Present the top patterns (5-10) as a triage table. For each pattern:
- Brief description of what it catches
- Frequency (how many times observed)
- Verbatim examples
- Coverage status (already covered / partially covered / not covered)
- Suggested check (regex, count, layout, etc.)

Ask the user to classify each pattern:
- **ADD** — write to user-overrides.yaml as a custom check
- **SKIP** — too niche or already implicit in defaults
- **PROMOTE** — this is universal enough to suggest as a default skill update (note for the maintainer)

### Step 4: Write user-overrides.yaml

For each ADD pattern, append to `~/.claude/skills/slide-inspector/user-overrides.yaml`:

```yaml
checks:
  - id: <unique-id>
    description: "<what this catches>"
    severity: critical | warning | suggestion
    type: regex | xml-attribute | count | layout | visual
    pattern: "<implementation detail or regex>"
    derived_from: "Found in N sessions, e.g. '<verbatim example>'"
    enabled: true
```

If the file doesn't exist, create it with frontmatter:

```yaml
# user-overrides.yaml — custom checks for slide-inspector
# Generated by calibration. Hand-edit freely.

overrides:
  density_word_threshold: 40   # presenter mode default

checks:
  - id: ...
```

### Step 5: Confirm

Show the user the new user-overrides.yaml and confirm the additions. Tell them they can hand-edit it anytime, and re-run calibration in the future to add more patterns.

## Loading user-overrides.yaml in inspections

When `inspect_slides.py` runs, it does NOT currently load user-overrides.yaml directly — that's handled at the SKILL.md / Claude-judgment layer. After running the programmatic analyzer, Claude reads `~/.claude/skills/slide-inspector/user-overrides.yaml` (if present) and applies each enabled check against the slide data, adding any matches to the issue report.

This keeps the Python script general-purpose and lets custom checks evolve via plain YAML without code changes.
